# مراجعة وحدتي المشاريع والتطبيقات وفق المعايير الدولية

## 1. نظرة عامة على البنية الحالية

### core.projects — جدول المشاريع
```
id, institution_id, project_code, title_ar, title_en, abstract_ar, abstract_en,
objectives, principal_investigator_id, research_category, risk_level, status_code,
start_date, expected_end_date
```
- العلاقات: مؤسسة، باحث رئيسي، فرق العمل، المواقع، مصادر التمويل، الكلمات المفتاحية، الوسوم، المرفقات
- التحكم بالنسخ (Versioning) عبر `core.project_versions` (snapshot_data jsonb)
- تتبع تاريخ الحالة عبر `core.project_status_history`

### core.applications — جدول التطبيقات
```
id, application_number, project_id, application_type (INITIAL|AMENDMENT|RENEWAL|EXPEDITED),
current_status, submission_date, submitted_by, priority_level, target_committee_id, remarks
```
- الأنواع: INITIAL, AMENDMENT, RENEWAL, EXPEDITED
- 11 حالة معتمدة في `reference.application_statuses`

### workflow — محرك سير العمل
- 9 حالات (states): DRAFT ← SUBMITTED ← INITIAL_REVIEW ← SCIENTIFIC_REVIEW ← ETHICAL_REVIEW ← COMMITTEE_REVIEW → APPROVED/REJECTED/RETURNED
- 14 انتقالة (transitions) مع أدوار وصلاحيات محددة لكل انتقالة
- متطلبات التعليق إلزامية في حالات الرفض أو الإعادة

### committee — لجان المراجعة
- `ethics_reviews` — مراجعة أخلاقية منفصلة عن `scientific_reviews`
- `review_forms` + `review_questions` — نموذج مراجعة ديناميكي (form builder)
- `review_assignments` — توزيع المراجعين مع `due_date` و `status_code`
- `review_answers` + `review_scores` — إجابات وتقييمات
- `review_comments` — تعليقات داخلية/خارجية
- `review_recommendations` — توصيات المراجعين
- `voting_sessions` + `votes` — جلسات التصويت
- `committee_meetings` + `agenda_items` + `meeting_minutes` + `quorum_logs` — إدارة الاجتماعات

## 3. الفجوات الحرجة (Critical Gaps)

### 3.1 ❌ الموافقة المستنيرة (Informed Consent)
لا يوجد جدول مخصص لنظام الموافقة المستنيرة. الـ CIOMS والـ ICH-GCP يتطلبان:
- **إصدارات الموافقة**: تاريخ، رقم إصدار، لغة
- **أنواع الموافقة**: كتابية، شفوية، موافقة ولي الأمر، موافقة الوصي
- **تتبع التوقيع**: موقع، تاريخ، شاهد
- **الموافقة على استخدام العينات البيولوجية** أو **البيانات الجينية**
- **سحب الموافقة**: تاريخ، سبب

**الحل المقترح:**
```sql
CREATE TABLE core.informed_consents (
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  application_id BIGINT NOT NULL REFERENCES core.applications(id),
  consent_version VARCHAR(50) NOT NULL,
  consent_type VARCHAR(100) NOT NULL,   -- WRITTEN, VERBAL, PARENTAL, GUARDIAN, LEGAL_REPRESENTATIVE
  consent_language VARCHAR(50) DEFAULT 'AR',
  document_id BIGINT REFERENCES documents.documents(id),
  participant_identifier VARCHAR(200),
  consent_date DATE,
  withdraw_date DATE,
  withdraw_reason TEXT,
  witness_name VARCHAR(300),
  witness_signature_document_id BIGINT REFERENCES documents.documents(id),
  is_active BOOLEAN DEFAULT true,
  created_at, updated_at, deleted_at, ...
);
```

### 3.2 ❌ سجل التدريب للأعضاء
لا يوجد جدول لتتبع تدريب أعضاء اللجنة. الـ WHO يطلب:
- تدريب أولي ودوري
- شهادات حضور ورش العمل
- تدريب على الـ GCP والـ Helsinki Declaration

### 3.3 ❌ الإفصاح المالي (Financial Disclosure)
لا يوجد هيكل للإفصاح المالي أو تضارب المصالح المالية
- مصدر التمويل موجود (`project_funding_sources`) لكن لا يوجد إفصاح عن المصالح المالية للأعضاء/الباحثين

### 3.4 ⚠️ تحليل المخاطر والفوائد
`risk_level` حقل نصي بسيط. الـ CIOMS يتطلب تحليل منظم:
- تحديد المخاطر (جسدية، نفسية، اجتماعية، قانونية)
- احتماليتها وشدتها
- خطة التخفيف لكل خطر
- المقارنة مع الفوائد المتوقعة

### 3.5 ⚠️ الإجراءات التشغيلية المعيارية (SOPs)
لا يوجد ربط بين SOPs ونماذج المراجعة (review_forms). مثلاً:
- SOP لمراجعة الطلبات الجديدة
- SOP للمراجعة السريعة
- SOP لمراجعة التعديلات
- يجب أن يحدد كل SOP أي form يستخدم

### 3.6 ⚠️ التسجيل العام للدراسة
لا يوجد حقل لرقم التسجيل العام (مثل ClinicalTrials.gov NCT أو Iranian Registry IRCT).
العديد من المجلات تطلب هذا كشرط للنشر.

## 4. نقاط القوة الرئيسية

### ✅ محرك سير العمل (Workflow Engine)
مصمم بشكل **مرن جداً** — يمكن إضافة حالات وانتقالات جديدة دون تغيير الكود. يتبع نمط State Machine مع:
- تعريف سير العمل في `workflow.workflows`
- حالات (`states`) محددة لكل سير
- انتقالات مع أدوار وصلاحيات
- SLA وتصعيد (`workflow_sla` + `workflow_escalations`)

### ✅ نماذج المراجعة الديناميكية (Form Builder)
`review_forms` + `review_questions` + `review_answers` توفر نظاماً:
- قابلاً للتخصيص حسب نوع التقديم أو اللجنة
- أسئلة بنوع مختلف (نص، اختيار، تقييم)
- ترتيب عرض وعلامات وزن لكل سؤال

### ✅ المسار المزدوج للمراجعة (Scientific + Ethical)
فصل المراجعة العلمية عن الأخلاقية — ممارسة مثالية حسب الـ CIOMS والـ WHO

### ✅ وحدة المراقبة (Monitoring)
نظام متكامل للمراقبة بعد الموافقة (post-approval monitoring):
- خطط مراقبة
- زيارات ميدانية
- تقارير امتثال
- متابعة الانحرافات والمخالفات
- إجراءات تصحيحية ووقائية

### ✅ التدقيق والتتبع (Audit)
`audit` schema مع:
- سجل لكل تغيير في أي جدول (trigger-based)
- hash ledger لاكتشاف التلاعب
- tracking entity changes

### ✅ الفصل بين الصلاحيات (Segregation of Duties)
```sql
security.segregation_rules  -- منع تضارب الأدوار
```

### ✅ حوكمة اللجان
```sql
committee.member_terms      -- فترات العضوية
committee.member_qualifications -- المؤهلات والشهادات
committee.member_conflicts  -- تضارب المصالح
committee.quorum_logs       -- النصاب القانوني
committee.voting_sessions   -- جلسات التصويت المقرونة
```

## 7. تحليل إضافي — مراجعة Lead Architect & Ethics Systems Auditor

### 7.1 ❗ Project ≠ Protocol — فجوة معمارية

النظام الحالي يدمج **المشروع** و **البروتوكول** في جدول واحد (`core.projects`). في الأنظمة الدولية، غالباً ما يكون الفصل كالتالي:

```
Research Project (مشروع)
  ├── Protocol V1 (بروتوكول)
  ├── Protocol V2
  ├── Protocol Amendment A
  └── Protocol Amendment B
```

**سيناريو واقعي لوزارة الصحة:**

```
COVID-19 Research Program (مشروع واحد)
  ├── Protocol V1: دراسة انتشار COVID-19 في صنعاء
  ├── Protocol V2: إضافة فرع عدن بعد التعديل
  ├── Protocol Amendment A: إضافة اختبار PCR
  └── Protocol Amendment B: تمديد المتابعة
```

**التأثير الحالي:** إذا كان `project == protocol` حالياً، هذا مقبول لـ RC1 لكنه سيسبب مشاكل مستقبلية عندما يكون للمشروع الواحد بروتوكولات متعددة أو تعديلات متوازية.

**الحل المقترح للمستقبل:**
- إضافة طبقة `core.protocols` بين `projects` و `applications`
- `application` ترتبط بـ `protocol` وليس بـ `project` مباشرة
- دعم إصدارات البروتوكول (`protocol_versions` مشابهة لـ `application_versions`)

### 7.2 ❗ Informed Consent — تصميم منفصل بثلاثة جداول

الحل المقترح الأصلي (جدول واحد `informed_consents`) مبسط جداً. المعيار الحديث يتعامل مع ثلاثة كيانات منفصلة:

| الكيان | الدور |
|--------|-------|
| `consent_templates` | قالب الموافقة (نص الموافقة الأساسي، لكل تطبيق) |
| `consent_versions` | إصدارات القالب (رقم الإصدار، تاريخ الاعتماد، اللغة) |
| `consent_records` | سجل موافقة كل مشارك (أي إصدار وقّع، تاريخ، شاهد، سحب) |

**مثال:**

```
Template: "الموافقة المستنيرة لدراسة COVID-19"
  └── Version 1.0 (AR) — 2026-01-15
  └── Version 1.0 (EN) — 2026-01-15
  └── Version 1.1 (AR) — 2026-03-01 (بعد تعديل البروتوكول)

Record: participant #42
  └── Version 1.0 (AR) — signed 2026-01-20
  └── Withdrawn 2026-04-01
```

### 7.3 ❗ اعتماد اللجنة (Committee Accreditation) — أهم من Training Records

الملف الأصلي ذكر `training_records` لكن **الأهم** هو **اعتماد اللجنة نفسها**:

```
committee_accreditations
  ├── accreditation_date
  ├── accreditation_expiry
  ├── accrediting_body
  ├── scope (مثل: البحوث الدوائية، الأجهزة الطبية)
  ├── status (ACTIVE, SUSPENDED, EXPIRED, REVOKED)
  ├── notes
  └── last_quorum_check (هل فقدت اللجنة النصاب؟)
```

في الأنظمة الوطنية، اللجنة تفقد اعتمادها إذا:
- فقدت النصاب القانوني
- انتهت صلاحية العضوية
- توقف العمل لأكثر من X أشهر

### 7.4 ❗ المراجعة الدورية المستمرة (Continuing Review)

وجود `application_type = 'RENEWAL'` ليس كافياً. المطلوب:

```
APPROVED
  ↓ (11 months)
Reminder → Continuing Review Notice
  ↓ (month 12)
Annual Continuing Review
  ├── APPROVED → تمديد سنة إضافية
  ├── CONDITIONAL → تعديلات مطلوبة
  └── NOT CONTINUED → إغلاق
```

**المكونات المطلوبة:**
1. **جدول `continuing_review_schedule`**: لكل تطبيق، تاريخ المراجعة القادمة، فترة السماح
2. **جدول `continuing_reviews`**: سجل كل مراجعة دورية (نتيجتها، توصياتها)
3. **خلفية (Scheduler)**: إشعار تلقائي قبل الموعد بشهر (نظام الـ workflow يدعم `workflow_schedulers`)

### 7.5 ❗ تحليل المخاطر المنظم — أولوية أعلى من Informed Consent

أتفق مع المراجعة الأصلية، لكن **أولوية تحليل المخاطر > الموافقة المستنيرة** لأن:

1. الموافقة المستنيرة يمكن إرفاقها كمستند (`documents.documents` مع entity_type='APPLICATION') مؤقتاً
2. تحليل المخاطر هو **جزء أساسي من قرار اللجنة نفسه** — بدونه، قرار اللجنة غير مكتمل معيارياً
3. `risk_level` بسيط جداً ولا يتناسب مع مستوى بقية النظام

**نموذج مقترح:**

```sql
project_risk_assessments (
  id, project_id,
  risk_category VARCHAR(100),    -- جسدي، نفسي، اجتماعي، قانوني، خصوصية
  risk_description TEXT NOT NULL,
  probability VARCHAR(20),       -- REMOTE, UNLIKELY, POSSIBLE, LIKELY, CERTAIN
  severity VARCHAR(20),          -- NEGLIGIBLE, MINOR, MODERATE, SEVERE, CRITICAL
  risk_score NUMERIC(5,2),       -- probability × severity (مثال: 3×4 = 12)
  mitigation_plan TEXT,
  residual_risk VARCHAR(20),     -- بعد التخفيف
  benefit_assessment TEXT,       -- مقارنة بالفائدة المرجوة
  is_acceptable BOOLEAN,
  assessed_by BIGINT REFERENCES users,
  assessed_at TIMESTAMPTZ
)
```

### 7.6 ⚠️ دراسات متعددة المواقع (Multi-Site)

لأن النظام يستهدف **وزارة الصحة اليمنية**، متوقع دراسات تمتد على:

```
صنعاء، عدن، تعز، الحديدة، حضرموت، ...
```

`project_sites` الحالي يدعم:
```
site_name, governorate, address, expected_participants
```

**لكن ينقصه:**
- `site_pi_id` → الباحث الرئيسي في كل موقع (موجود في `project_site_investigators` لكن ليس بالضرورة لكل موقع)
- `site_approval_status` → هل وافقت إدارة الموقع؟
- `site_approval_date`
- `site_status` → ACTIVE، INACTIVE، ON_HOLD
- `site_local_ethics_approval` → موافقة اللجنة المحلية (إن وجدت)

الحل: إضافة أعمدة `site_pi_id` و `site_approval_status` و `site_status` إلى `project_sites`.

### 7.7 ⚠️ مشاركة المجتمع (Community Engagement)

CIOMS 2016 يعطي أهمية متزايدة لـ **مشاركة المجتمع** خاصة في:
- الصحة العامة
- الأمراض السارية
- الدراسات المجتمعية في المناطق الريفية

مقترح:
- إضافة `community_engagement_plan` (نص) في جدول المشاريع
- أو جدول `community_consultations` للتشاورات المجتمعية

### 7.8 ✅ أقوى نقطة في النظام — محرك سير العمل (Workflow Engine)

أقوى جزء معماري في النظام برمته هو **Workflow Engine**:

```
workflow.workflows
  ├── workflow_states
  ├── workflow_transitions (مع قواعد الأدوار)
  ├── workflow_instances
  ├── workflow_sla
  ├── workflow_escalations
  ├── workflow_schedulers
  └── workflow_triggers
```

وجود State Machine مع SLA وتصعيد وجدولة **يجعل النظام قابلًا للتوسع لسنوات** — هذه من أفضل القرارات المعمارية التي اتخذت في هذا المشروع.
