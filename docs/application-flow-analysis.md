# تحليل دورة حياة الطلب وتدفق المسودة

## المشكلة: دورة الطلب مفقودة للإنشاء على مراحل

**الوضع السابق**: إنشاء الطلب يتم في خطوة واحدة (POST مباشر) مع بدء سير العمل فوراً دون إعطاء المستخدم فرصة لحفظ مسودة أو رفع مرفقات قبل التقديم.

## الفجوات المحددة

### قاعدة البيانات
- جدول `core.applications` يدعم `current_status DEFAULT 'DRAFT'` أصلاً ✅
- يوجد عمود `submission_date` nullable ✅
- يوجد عمود `priority_level`, `remarks` ✅
- البنية جاهزة للمسودات — المشكلة في طبقة الخدمة فقط

### الباك-إند (الخدمات والمسارات)
| المكون | قبل الإصلاح | بعد الإصلاح |
|--------|-------------|-------------|
| `POST /core/applications` | ينشئ + يبدأ workflow فوراً | يدعم `?save_as_draft=true` يتخطى workflow |
| `PUT /core/applications/:id` | **غير موجود** | يحدّث الطلبات بحالة DRAFT فقط |
| `POST /core/applications/:id/submit` | **غير موجود** | يغيّر الحالة إلى SUBMITTED ويبدأ workflow |
| `updateDraft()` في الخدمة | **غير موجود** | يتحقق من ملكية المسودة وحالتها قبل التحديث |
| `submitDraft()` في الخدمة | **غير موجود** | يتحقق من الحالة DRAFT للمالك فقط |

### الواجهة الأمامية (الصفحات)
| الصفحة | قبل الإصلاح | بعد الإصلاح |
|--------|-------------|-------------|
| `Create.tsx` | نموذج بسيط بثلاثة حقول فقط | **معالج بأربع خطوات**: 1-المعلومات الأساسية، 2-تفاصيل البحث، 3-المرفقات، 4-المراجعة والتقديم |
| `Edit.tsx` | **غير موجودة** | صفحة جديدة لتعديل المسودات (نفس المعالج مع تعبئة مسبقة) |
| `List.tsx` | جدول عرض فقط | أزرار تعديل وتقديم للطلبات بحالة DRAFT |
| `Detail.tsx` | عرض فقط + workflow/قرارات | زر تعديل للمسودات |
| `App.tsx` (المسارات) | `/create` و `/:id` فقط | إضافة مسار `/:id/edit` |

### SDK ونماذج البيانات
| المكون | التغيير |
|--------|---------|
| `Application` type | إضافة `current_status`, `application_type`, `committee_name` وغيرها |
| `CreateApplicationRequest` | إضافة `save_as_draft` اختياري |
| SDK `applications` | إضافة `updateDraft()`, `submitDraft()` |

## تدفق المستخدم الجديد

```
إنشاء مشروع ← إنشاء طلب
                │
                ├─► حفظ كمسودة (حالة DRAFT، بدون workflow)
                │       │
                │       ├─► تعديل المسودة (تغيير النوع/اللجنة/الملاحظات/المرفقات)
                │       │
                │       └─► تقديم المسودة → حالة SUBMITTED → بدء workflow
                │
                └─► تقديم مباشر → حالة SUBMITTED → بدء workflow (كما كان)
```

## المرفقات

تم دمج رفع المستندات في تدفق إنشاء الطلب:
1. في الخطوة الثالثة من المعالج، يختار المستخدم ملفات ويرفعها مؤقتاً
2. بعد إنشاء الطلب (مسودة أو تقديم)، تُرفع الملفات إلى الخادم مع `entity_id` الصحيح
3. في صفحة التعديل، يمكن عرض المرفقات الموجودة وحذفها وإضافة جديدة

## بنية المرفقات

جدول `documents.documents`:
- `entity_type = 'Application'`
- `entity_id = {id}`
- يدعم التحميل عبر `POST /documents` (multipart/form-data)

## ملاحظات فنية

- `AuditableRepository` يضبط `app.user_id` تلقائياً → RLS يعمل للمسودات أيضاً
- `PUT /core/applications/:id` يسمح فقط للمالك أو المشرف (SUPER_ADMIN/ETHICS_ADMIN) بتعديل المسودات
- `POST /core/applications/:id/submit` يسمح فقط للمالك بتقديم المسودة
- المسودات لا تظهر في workflow timeline لأنها لم تبدأ workflow بعد
- مسار `/applications/:id/edit` مسجل في `App.tsx` قبل `/:id` لتجنب تعارض المسارات
